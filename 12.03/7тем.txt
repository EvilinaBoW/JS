Отладка в браузере
DevTools — это встроенные в браузер инструменты разработчика. В Chrome вкладка Sources открывается клавишей F12 (или Cmd+Opt+I) с последующим выбором вкладки в верхней панели.

Оператор debugger останавливает выполнение кода в той строке, где он написан, если открыты инструменты разработчика. Полезен, когда сложно найти нужный файл в дереве исходников вручную.

Чтобы поставить breakpoint, нужно кликнуть на номер строки в панели Sources. При остановке выполнение программы замирает, позволяя изучить текущие значения переменных и состояние стека.

Step Over — переход к следующей строке без захода внутрь функций. Step Into — вход внутрь вызываемой функции. Step Out — завершение текущей функции и возврат к месту её вызова.

При остановке значения видны при наведении курсора на переменную в коде, а также в правой панели в секции Scope.

Watch — панель для отслеживания значений конкретных выражений. Добавить переменную можно нажатием на + в секции Watch и вводом её имени.

Conditional breakpoint ставится правой кнопкой мыши на номер строки -> "Add conditional breakpoint". Условие i > 5 заставит код остановиться только при истинности этого выражения.

Вкладка Console позволяет вводить JS-кода, который имеет доступ ко всем переменным в текущей области видимости на момент паузы.

Call Stack показывает цепочку вызовов функций, которые привели к текущей строке. Верхняя строка — текущая функция, ниже — те, что её вызвали.

console.table([{a:1, b:2}, {a:3, b:4}]) выводит данные в виде интерактивной таблицы.

Logpoint (правой кнопкой на номер строки -> Add logpoint) выводит сообщение в консоль без остановки выполнения кода и без изменения исходного файла.

Асинхронный стек включается галочкой Async в панели Call Stack. Он позволяет видеть вызовы, предшествующие асинхронному переходу (например, выполнение внутри setTimeout).

Blackboxing позволяет игнорировать выбранные скрипты (например, библиотеки вроде jQuery или React) при отладке, чтобы Step Into не заходил в их внутренний код.

Вкладка Performance записывает работу страницы. Flame graph — это визуальное представление дерева вызовов, где ширина блока соответствует времени выполнения функции.

Heap Snapshot делает снимок всей памяти. Сравнивая два снимка, можно увидеть объекты, которые не были удалены сборщиком мусора, что указывает на утечку.

Советы по стилю кода
Code style — набор правил оформления кода. Единый стиль делает код предсказуемым и читаемым для всех участников команды.

camelCase (переменные, функции), snake_case (иногда константы или API), PascalCase (классы). В JS стандарт — camelCase и PascalCase.

ESLint — статический анализатор, который находит ошибки и нарушения правил оформления кода до его запуска.

Prettier — автоматический форматировщик. В отличие от ESLint, он не ищет логические ошибки, а только переписывает код согласно правилам отступов и скобок.

Длинные строки трудно читать. Обычно используют стандарт 80 или 100 символов.

EditorConfig синхронизирует базовые настройки (отступы, кодировка) между разными редакторами. Пример .editorconfig:

Ini, TOML
root = true
[*]
indent_style = space
indent_size = 2
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true
Функция должна выполнять действие только одного уровня сложности.

JavaScript
// Плохо: считает и сохраняет
function process() { const res = 1 + 2; localStorage.setItem('res', res); }
// Хорошо
function sum(a, b) { return a + b; }
function save(val) { localStorage.setItem('res', val); }
Magic numbers — числа с неясным смыслом. Их нужно заменять именованными константами: const SECONDS_IN_DAY = 86400;.

DRY — не дублируй код. Если логика повторяется, её нужно вынести в отдельную функцию или класс.

KISS — делай проще. Не используй сложные конструкции там, где достаточно базового цикла или условия.

Глубокая вложенность (пирамида) затрудняет чтение. Решается использованием Guard Clauses (ранний выход) или выносом логики в функции.

Булевы переменные должны отвечать на вопрос "да/нет". Хорошо: isActive, hasUser, canClick. Плохо: status, user_exists.

no-shadow запрещает создавать переменные с именами, которые уже есть во внешней области видимости. Это предотвращает путаницу, какую переменную мы сейчас используем.

Early return — возврат из функции сразу, как только стало известно, что дальнейшее выполнение не требуется.

JavaScript
function check(age) {
  if (age < 18) return false;
  // логика для взрослых
  return true;
}
Настройка: npm install -D eslint prettier husky lint-staged, инициализация конфигураций и добавление скрипта lint-staged в package.json для запуска проверок на хук pre-commit.

Комментарии
// — до конца строки. /* */ — для блоков текста или внутри выражений.

JSDoc — стандарт документирования кода.

JavaScript
/**
 * Считает сумму
 * @param {number} a
 * @param {number} b
 * @returns {number}
 */
function sum(a, b) { return a + b; }
@param описывает тип и назначение аргумента, @returns описывает то, что функция возвращает.

Комментарий "что" дублирует код. Комментарий "почему" объясняет бизнес-логику или неочевидные решения (например, обход бага браузера).

// TODO: текст — пометка о том, что нужно доделать или исправить. Большинство IDE подсвечивают их в отдельном списке.

Комментирование i++ // инкремент бесполезно и замусоривает код.

Self-documenting code — код, который понятен по именам функций и переменных. Вместо const d = 86400; // секунд в дне использовать const SECONDS_IN_DAY = 86400;.

@deprecated — помечает устаревший код, @throws — какие ошибки выбрасывает, @example — пример использования.

В Review комментарии фиксируют договоренности. Commenting contract — описание ожидаемого поведения интерфейса через комментарии.

Генерация: установить пакет (например, jsdoc), создать конфиг и запустить команду jsdoc -c conf.json.

Literate programming — концепция, где код является дополнением к тексту на естественном языке. Влияет на стремление писать максимально "человекочитаемый" код.

Ниндзя-код
Ниндзя-код — ироничное название для запутанного кода. Он мешает поддержке, тестированию и масштабированию проекта.

x, t, d допустимы только в очень коротких циклах или математических формулах. В остальных случаях они лишают переменную смысла.

Сокращения вроде brsr заставляют разработчика тратить время на "декодирование" смысла.

Переиспользование переменной приводит к ошибкам, так как её тип или смысл могут внезапно измениться в середине функции.

"Умные" однострочники (цепочки тернарных операторов) экономят место, но их невозможно отлаживать по шагам.

Префикс _ обычно означает "private", но в обычном JS это лишь договоренность. Бессмысленное использование путает.

Изменение встроенных прототипов может привести к конфликтам с будущими стандартами JS или другими библиотеками.

Побочные эффекты в условиях: if (result = getData()). Это легко принять за опечатку (вместо ==) и трудно заметить при чтении.

Возврат разных типов (число или объект) заставляет вызывающий код делать лишние проверки if. Лучше возвращать объект с предсказуемой структурой.

В коде function f(a,b,c,d){return a?b?c:d:b}: неясные имена, избыточные вложенные тернарники, отсутствие читаемой логики.

Минификация уменьшает размер файла для загрузки. Обфускация делает код нечитаемым для человека с целью защиты интеллектуальной собственности.

Автотестирование с Mocha
Автотестирование — проверка кода с помощью другого кода. Экономит время на ручном тестировании и предотвращает регрессии.

Mocha — это тестовый фреймворк (запускалка), а Jest — это "всё в одном" (включая матчеры и моки).

describe группирует тесты, it — описывает конкретный тест.

JavaScript
describe('Math', () => {
  it('should sum', () => { /* test */ });
});
equal проверяет равенство (==), strictEqual проверяет тип и значение (===).

Chai — библиотека утверждений. Подключается через require('chai') или тегом <script>.

expect(sum(2, 3)).to.equal(5);

BDD (Behavior Driven Development) описывает поведение системы на естественном языке: "оно должно делать то-то".

before/after запускаются один раз на блок, beforeEach/afterEach — перед и после каждого it.

TDD: сначала пишем падающий тест (Red), затем минимальный код для его прохождения (Green), затем улучшаем код (Refactor).

Тесты для factorial(n): factorial(0) -> 1, factorial(1) -> 1, factorial(-1) -> throw Error.

Test coverage показывает процент строк кода, затронутых тестами. Измеряется инструментами типа istanbul (nyc).

Stub заменяет функцию и возвращает заданное значение. Mock проверяет, как именно вызывалась функция (с какими аргументами).

Асинхронность: вернуть Promise из it или использовать аргумент done() и вызвать его в конце.

.only запускает только этот тест/группу, .skip — пропускает их.

expect(() => func(badArg)).to.throw();

Test suite — набор тестов. Обычно файлы именуются *.test.js и лежат в папке tests/ или рядом с исходником.

Snapshot testing сравнивает текущий результат работы (например, HTML) с ранее сохраненным эталоном.

Для тестов с побочными эффектами используют заглушки. Dependency injection — это передача зависимостей (например, клиента БД) в функцию извне, чтобы в тестах подсунуть фейковый клиент.

Property-based testing генерирует сотни случайных входных данных для проверки общих свойств функции.

В GitHub Actions создается файл .yml, где указывается команда npm test, запускаемая при событии push.

Полифилы
Полифил — код, который реализует современную функциональность в старых браузерах.

Транспайлер (Babel) меняет синтаксис (напр. () => на function), а полифил добавляет отсутствующие методы (Array.prototype.includes).

Feature detection: if (!Array.prototype.includes) { ... }.

Полифил для includes:

JavaScript
if (!Array.prototype.includes) {
  Array.prototype.includes = function(search) {
    return this.indexOf(search) !== -1;
  };
}
core-js — самая большая библиотека полифилов. Подключается через import "core-js/stable";.

@babel/preset-env анализирует список целевых браузеров и автоматически решает, какие полифилы из core-js нужно импортировать.

Полифил для flat:

JavaScript
if (!Array.prototype.flat) {
  Array.prototype.flat = function() {
    return this.reduce((acc, val) => acc.concat(val), []);
  };
}
Полифил для Object.assign:

JavaScript
if (typeof Object.assign !== 'function') {
  Object.assign = function(target, ...sources) {
    sources.forEach(source => {
      if (source) Object.keys(source).forEach(key => target[key] = source[key]);
    });
    return target;
  };
}
browserslist — файл .browserslistrc, где указываются поддерживаемые браузеры (например, > 0.25%, not dead). Инструменты используют его для оптимизации полифилов.

Полифил для Promise требует реализации: 1. Состояния (pending/fulfilled/rejected). 2. Обработки .then(). 3. Статических методов resolve/reject/all.

useBuiltIns: usage заставляет Babel добавлять import только тех полифилов, которые реально используются в коде проекта.

Shim — более широкое понятие, библиотека, которая меняет API или поведение. Polyfill — это shim для соответствия стандарту API.

Полифил для flatMap:

JavaScript
if (!Array.prototype.flatMap) {
  Array.prototype.flatMap = function(cb) {
    return this.map(cb).flat();
  };
}
Полифил для globalThis:

JavaScript
(function() {
  if (typeof globalThis !== 'undefined') return;
  Object.defineProperty(Object.prototype, '__magic__', {
    get: function() { return this; },
    configurable: true
  });
  __magic__.globalThis = __magic__;
  delete Object.prototype.__magic__;
}());
Scoped polyfill — использование локальной функции-заменителя (или библиотеки Ponyfill) вместо расширения глобального объекта, чтобы не конфликтовать с чужим кодом.
